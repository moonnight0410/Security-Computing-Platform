# 项目协作文档

## 1. 文档目的

本文档用于统一本项目的协作方式，降低多人并行开发时的沟通成本，避免对本地数据、规则包、审计链和输出流程造成误操作。

适用范围：

- 后端 FastAPI 开发
- 前端 React + Vite 开发
- 规则包中心相关功能开发
- 本地运行、联调、测试、验收
- `docs/` 阶段文档补充与交付整理

当前项目阶段：`Stage 9`

当前阶段主题：规则包中心、修订快照、编辑治理。

---

## 2. 项目概览

本项目是一个“本域数据不出域”的单机系统，核心目标是在本地受控环境内完成数据导入、字段映射、规则包管理、任务执行、审计留痕、安全输出和归档验签。

当前代码形态为前后端分离：

- 后端：`FastAPI`
- 前端：`React 19 + Vite + TypeScript`
- 本地状态：`workspace/config/app-state.sqlite3`
- 审计链：`workspace/audit/audit-log.jsonl`
- 输出目录：`workspace/exports/`
- 归档目录：`workspace/archives/`

项目当前已实现的主线能力包括：

- 数据集导入与字段 profiling
- 字段映射配置
- 规则包创建、验签、审批
- Stage 9 规则包中心与修订历史
- 任务创建与本地执行
- 结果查看、结论声明复核
- 输出申请、输出审批、输出包落盘
- 归档封存与验签
- 审计链写入与完整性校验

## 3. 目录职责

```text
backend/    后端接口、服务逻辑、数据模型、单元测试
frontend/   前端页面、API 调用、类型定义、样式
docs/       阶段文档、设计说明、交付说明
scripts/    本地启动、停止、检查脚本
workspace/  运行态数据、日志、导出、归档、审计、本地状态
```

建议按以下边界协作：

- `backend/app/main.py`
  负责 API 路由编排和接口入口，不承载过重业务细节。
- `backend/app/services/`
  负责执行、导出、审计、规则包、归档、签名等核心逻辑。
- `backend/app/models/`
  负责请求与响应模型、结构定义。
- `frontend/src/App.tsx`
  负责工作台主流程页面。
- `frontend/src/RulePackageCenter.tsx`
  负责 Stage 9 规则包中心、修订编辑、审批治理。
- `frontend/src/api.ts`
  负责前端与后端接口契约。
- `backend/tests/`
  负责后端核心能力回归验证。

## 4. 协作角色建议

推荐最小协作角色如下：

- 产品/需求负责人
  负责确认 Stage 目标、边界和验收口径。
- 后端负责人
  负责接口、服务逻辑、状态持久化、审计链、安全输出。
- 前端负责人
  负责页面交互、接口接入、状态展示、联调体验。
- 测试/验收负责人
  负责脚本检查、关键流程回归、文档核验。
- 文档负责人
  负责 `README.md`、`docs/` 以及阶段完成报告同步更新。

如果是两人协作，建议按下面方式分工：

- A 负责 `backend/`、`scripts/`、`backend/tests/`
- B 负责 `frontend/`、`docs/`、联调验收记录

如果是三人协作，建议按下面方式分工：

- A 负责后端接口与服务层
- B 负责前端页面与交互
- C 负责测试、文档、验收和交付整理

---

## 5. 协作原则

- 所有功能默认围绕“本地运行、本域留存、不直接输出原始数据”展开。
- 改动前先确认影响的是工作台主流程还是规则包中心流程。
- 改动 `workspace/` 相关读写逻辑时，必须同步考虑审计记录是否完整。
- 改动输出、归档、签名、审批逻辑时，必须补测试或至少补脚本验证。
- 不直接修改历史阶段文档结论，新增内容优先写入新的 `docs/` 文档。
- 一次提交尽量只解决一个清晰问题，避免前后端与文档完全混杂。

## 6. 本地开发基线

### 6.1 后端环境

依赖文件：

- `backend/requirements.txt`

初始化方式：

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\pip install -r requirements.txt
```

手动启动：

```powershell
cd backend
.\.venv\Scripts\uvicorn app.main:app --host 127.0.0.1 --port 8000
```

脚本启动：

```powershell
.\scripts\start-backend.ps1
```

健康检查：

```text
http://127.0.0.1:8000/api/health
```

### 6.2 前端环境

依赖文件：

- `frontend/package.json`

安装与启动：

```powershell
cd frontend
npm install
npm run dev
```

脚本启动：

```powershell
.\scripts\start-frontend.ps1
```

访问地址：

```text
http://127.0.0.1:5173
```

### 6.3 停止本地服务

```powershell
.\scripts\stop-local.ps1
```

## 7. 标准协作流程

### 7.1 开发前

开始开发前，至少确认以下事项：

- 当前目标属于哪个 Stage 能力范围
- 是否影响 `App.tsx` 工作台主链路
- 是否影响 `RulePackageCenter.tsx` 的修订编辑链路
- 是否影响 `backend/app/main.py` 的接口契约
- 是否影响 `workspace/` 中的状态、审计、导出、归档
- 是否需要补充 `docs/` 说明文档

### 7.2 开发中

推荐工作顺序：

1. 先明确接口或页面边界。
2. 再修改后端服务或前端交互。
3. 完成后执行本地检查。
4. 最后补齐文档和验收说明。

多人并行时建议遵循：

- 前端先锁定接口字段名，避免联调时反复改 `types.ts`。
- 后端新增字段时，同步更新 `frontend/src/types.ts` 与 `frontend/src/api.ts`。
- 修改任务执行结果结构时，前后端必须同轮联调。
- 修改规则包修订结构时，必须同时检查历史快照读取是否兼容。

### 7.3 开发后

提交前至少完成：

- 本地脚本检查
- 关键流程手工验证
- 相关文档更新
- 说明本次改动影响范围

## 8. 文件改动边界建议

为避免多人冲突，建议按以下方式控制写入范围。

后端常见改动边界：

- 新增接口：优先改 `backend/app/main.py`
- 新增业务逻辑：优先放到 `backend/app/services/`
- 新增结构定义：放到 `backend/app/models/schemas.py`
- 新增状态集合或迁移逻辑：检查 `backend/app/services/storage.py`
- 新增安全输出逻辑：检查 `backend/app/services/exports.py`
- 新增审计逻辑：检查 `backend/app/services/audit.py`
- 新增归档验签逻辑：检查 `backend/app/services/archives.py`

前端常见改动边界：

- 工作台流程：改 `frontend/src/App.tsx`
- 规则包中心：改 `frontend/src/RulePackageCenter.tsx`
- 接口请求：改 `frontend/src/api.ts`
- 类型契约：改 `frontend/src/types.ts`
- 视觉样式：改 `frontend/src/styles.css`

文档常见改动边界：

- 阶段设计和完成报告：写入 `docs/`
- 总览说明：写入 `README.md`
- 协作约定、交付流程：优先补充本文档

## 9. Stage 9 协作重点

当前阶段最容易发生冲突的区域是规则包中心。

相关文件：

- `frontend/src/RulePackageCenter.tsx`
- `backend/app/services/rule_packages.py`
- `backend/app/services/rule_signatures.py`
- `backend/tests/test_rule_package_revisions.py`
- `backend/tests/test_rule_signatures.py`

协作时必须注意：

- 规则包不是直接覆盖编辑，而是通过修订快照推进。
- 草稿保存、提交验签、审批通过、废弃处理是不同状态流转。
- 已审批规则包如果被再次编辑，需要确认修订链路是否仍然可追溯。
- 改动签名字段、签名引用、验签状态时，必须同步验证前端展示和后端校验。
- 删除规则包前，需要确认是否被任务引用。

建议单人独占修改以下组合，避免冲突：

- `RulePackageCenter.tsx` + `frontend/src/types.ts`
- `rule_packages.py` + `schemas.py`
- 规则包相关测试文件

## 10. 联调与验证要求

### 10.1 推荐检查脚本

Stage 9 检查脚本：

```powershell
.\scripts\run-stage9-checks.ps1
```

脚本内容当前覆盖：

- 后端 `compileall`
- 后端 `unittest`
- 前端 `tsc --noEmit`
- 前端 `npm run build`

### 10.2 关键联调链路

每次较大改动后，建议至少手动走通以下链路：

1. 上传数据集
2. 保存字段映射
3. 选择已审批规则包创建任务
4. 执行任务并查看结果
5. 触发输出申请与审批
6. 落盘输出文件
7. 创建归档并执行验签
8. 检查审计链校验结果

### 10.3 必查文件

联调异常时优先检查：

- `workspace/logs/backend.err.log`
- `workspace/logs/frontend.err.log`
- `workspace/audit/audit-log.jsonl`
- `workspace/config/app-state.sqlite3`

## 11. 数据与安全协作约束

本项目不是普通 CRUD 项目，协作时必须显式遵守以下限制：

- 不将原始数据导出到受控目录之外。
- 不绕过输出审批链路直接生成外发文件。
- 不跳过审计写入逻辑。
- 不随意清空 `workspace/` 中的运行态数据。
- 不在测试数据中混入真实敏感信息。
- 不用前端展示替代后端安全校验。

涉及以下功能时，必须重点复核：

- `output_policy`
- `aggregate_threshold`
- `aggregate_group_by`
- 结论声明审批
- 输出审批
- 归档签名与验签
- 审计链完整性

## 12. 文档同步规则

以下情况应同步更新文档：

- 新增或删除 API
- 调整任务执行结果结构
- 调整规则包状态机或修订机制
- 调整输出审批、归档或验签流程
- 调整本地启动、停止、检查脚本

建议文档更新优先级：

1. `README.md`
2. 对应 Stage 说明文档
3. 本协作文档

## 15. 推荐协作方式

日常协作建议采用“接口先对齐、改动分区、脚本收口、文档补齐”的方式推进。

推荐节奏：

1. 先在需求层明确本次只动哪一条主线。
2. 再划分前端、后端、测试、文档责任。
3. 开发中保持 `api.ts` 和 `types.ts` 与后端契约同步。
4. 完成后统一执行 `run-stage9-checks.ps1`。
5. 最后由文档负责人补齐 `docs/` 和验收说明。

如果后续进入 `Stage 10`，建议在本文档基础上继续扩展“分支策略、测试基线、版本发布、问题分级响应”四部分，而不是重新另写一套协作规范。
