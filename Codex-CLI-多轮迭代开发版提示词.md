# Codex CLI 多轮迭代开发版提示词

你现在进入一个**分阶段、多轮迭代的软件开发模式**。你的任务不是一次性把所有代码铺开，而是严格按照阶段推进，在每一轮只完成当前阶段要求的内容，并在阶段末尾停止，等待下一条指令。

## 一、项目背景

请围绕以下目标开展设计与开发：

构建一个用于**政府部门联查数据计算**的单机系统，要求支持**前端 + 后端**完整交付，能够在**单机离线环境**中部署和运行，不依赖外网服务。

系统核心目标包括：

1. 支持多部门数据导入、管理与计算。
2. 支持通过**算子库**对数据进行多种规则计算、组合计算和结果输出。
3. 支持在不暴露原始敏感标识的前提下进行**去标识匹配**。
4. 支持基于统一规则生成**主键 / 子键派生体系**，用于跨部门数据关联。
5. 支持全过程留痕、结果可追溯、规则可解释。

## 二、必须遵守的事实边界

你必须严格遵守以下原则：

### 1. 只基于已知事实输出
- 只能根据我明确给出的需求、约束和上下文进行设计。
- 不得擅自虚构政策细节、行业规范、字段定义、接口来源、数据库现状、部署资源或已有系统能力。
- 如果某部分缺少事实依据，你必须在输出中明确标记为“待确认事项”“假设项”或“可选方案”，不能把猜测写成既定事实。

### 2. 不得越阶段执行
- 每一轮只做当前阶段要求的工作。
- 不得提前开始下阶段的代码实现、数据库建表、接口编写、页面开发或测试执行，除非我明确要求。

### 3. 输出必须为工程可落地内容
- 你的输出要面向真实开发落地，而不是泛泛而谈。
- 文档、方案、接口、模块划分、表结构建议、目录结构建议都要清晰、可执行、可继续迭代。

## 三、系统硬约束

在所有阶段中，下面这些约束都必须持续有效：

1. **部署模式**：仅考虑**单机离线部署**。
2. **系统形态**：必须包含**前端 + 后端**，不是纯脚本工具。
3. **数据处理重点**：必须覆盖**算子库设计**。
4. **数据关联重点**：必须覆盖**去标识匹配机制**。
5. **键体系重点**：必须覆盖**主键 / 子键派生规则设计**。
6. **安全与合规表达**：只做通用工程设计，不捏造未给出的政策条文。
7. **可扩展性**：方案要支持未来新增部门、新增算子、新增匹配规则。

## 四、开发阶段定义

你必须严格按以下阶段工作：

### Stage 0，设计文档阶段
本阶段只输出设计，不写实现代码。

本阶段目标：
- 梳理需求边界
- 识别已知事实、未知事项、关键假设
- 给出系统总体架构方案
- 给出前后端模块划分
- 给出核心数据流设计
- 给出算子库设计思路
- 给出去标识匹配方案思路
- 给出主键 / 子键派生方案思路
- 给出部署方案草案
- 给出风险点与待确认问题清单
- 给出后续 Stage 1 的实施建议

本阶段输出建议结构：
1. 项目目标
2. 已知约束
3. 事实边界与假设项
4. 总体架构
5. 前端设计
6. 后端设计
7. 数据模型建议
8. 算子库设计建议
9. 去标识匹配设计建议
10. 主键 / 子键派生设计建议
11. 单机离线部署建议
12. 风险与待确认项
13. Stage 1 开发范围建议

### Stage 1，基础骨架实现阶段
只有在我明确要求进入 Stage 1 后，才允许执行。

本阶段可做：
- 初始化项目目录
- 创建前后端工程骨架
- 搭建基础配置
- 搭建核心模块空实现
- 搭建基础页面框架
- 搭建基础 API 路由框架
- 不必一次写完全部业务

### Stage 2，核心能力实现阶段
只有在我明确要求进入 Stage 2 后，才允许执行。

本阶段可做：
- 数据导入管理
- 算子库核心实现
- 去标识匹配核心实现
- 主键 / 子键派生核心实现
- 核心结果展示与查询

### Stage 3，完善与验证阶段
只有在我明确要求进入 Stage 3 后，才允许执行。

本阶段可做：
- 补测试
- 补错误处理
- 补日志与审计留痕
- 完善部署脚本
- 完善文档
- 做最终收口

## 五、每一轮的执行规则

每次收到任务后，你都必须遵循以下流程：

1. 先判断当前属于哪个阶段。
2. 只输出该阶段允许的内容。
3. 如果发现存在关键信息缺失，先在结果中列出，不要自行编造。
4. 如果我没有明确让你进入下一阶段，你必须在本轮输出末尾停止。
5. 在末尾明确写出：
   - 当前完成的阶段
   - 已产出内容
   - 下一阶段可做事项
   - 需要我确认的事项

## 六、输出风格要求

1. 使用中文。
2. 结构清晰，标题分层明确。
3. 尽量使用表格、清单、分点说明，便于后续直接继续开发。
4. 不写空话，不写泛化套话。
5. 如果给出技术方案，需说明为什么这样设计。
6. 如果存在多个可行方案，优先给出**最简单、最稳妥、最适合单机离线部署**的方案，并说明备选方案。

## 七、首轮任务指令

现在请你先执行 **Stage 0，设计文档阶段**，并严格遵守以下要求：

- 只输出设计文档。
- 不要生成代码。
- 不要创建数据库脚本。
- 不要创建前端页面代码。
- 不要创建后端接口代码。
- 不要擅自进入 Stage 1。
- 输出完成后必须停止，等待我的下一条“进入 Stage 1”或其他后续指令。

## 八、最终停止要求

当你完成本轮 Stage 0 输出后，最后必须附上一段简短的阶段总结，格式如下：

```markdown
当前阶段：Stage 0，设计文档阶段
本轮产出：<列出本轮完成内容>
待确认事项：<列出待确认项，如无则写“无”>
下一步建议：等待用户指令，确认是否进入 Stage 1
```

现在开始执行。
