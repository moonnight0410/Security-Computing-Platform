# Stage 0 设计文档：本域数据不出域约束版

## 1. 背景更新

新增硬约束：

- 所有数据只能在本域内处理，不能出域。
- 原始数据不能给对方。
- 加密后的数据、哈希后的数据、去标识键、布隆过滤器、密文集合等也不能给对方。
- 不能以任何形式向对方提供自己的数据。

该约束会改变系统定位：系统不能设计为“跨域交换数据后联查”，也不能设计为“交换去标识数据后匹配”。系统应调整为“规则、问题、算子包可流转；数据、派生键、计算中间值不出本域；各域独立本地计算；跨域只交换经确认的不含本域数据的状态或结论”。

## 2. 项目目标

构建一个用于政府部门联查数据计算的单机离线系统，每个部门或数据域各自部署一套。本系统只处理本域数据，不向其他域输出原始数据、加密数据、哈希数据、去标识数据或可用于反推本域数据的中间结果。

系统目标调整为：

- 支持本域数据导入、字段画像、规则计算和审计留痕。
- 支持导入不含数据的联查规则包、事项包或算子包。
- 支持在本域内完成主键 / 子键派生，但派生结果不得出域。
- 支持在本域内完成去标识处理，但去标识结果不得出域。
- 支持生成本域内部结果、执行回执、人工复核材料或经授权的结论声明。
- 支持全过程留痕，证明“数据未出域、规则如何执行、谁执行、何时执行”。

## 3. 已知约束

| 约束 | 设计影响 |
| --- | --- |
| 数据不能出域 | 不能设计数据交换、数据上传、跨域数据汇聚 |
| 加密数据也不能出域 | 不能使用密文交换、PSI、MPC 输入交换、哈希集合交换作为方案 |
| 任何形式的本域数据都不能给对方 | 去标识键、派生键、样本、明细、可还原统计均不得跨域输出 |
| 系统必须单机离线部署 | 每个域独立部署，本地运行、本地存储 |
| 必须包含前端 + 后端 | 不是纯脚本，需有本地操作界面和本地 API |
| 必须覆盖算子库 | 算子在本域内执行，算子定义可流转，算子输入输出不出域 |
| 必须覆盖去标识匹配 | 仅限本域内去标识匹配，不作为跨域交换机制 |
| 必须覆盖主键 / 子键派生 | 仅限本域内部关联和追溯，不跨域输出 |

## 4. 事实边界与假设项

### 4.1 明确不能实现的能力

在当前约束下，以下能力不能实现或不应设计为系统能力：

- 不能把 A 域数据给 B 域进行匹配。
- 不能把 A 域数据加密后给 B 域。
- 不能把 A 域哈希主键、去标识主键、布隆过滤器、签名集合给 B 域。
- 不能在一个中心节点汇聚多域数据后计算。
- 不能做依赖双方密文输入交换的 PSI 或安全多方计算。
- 不能输出对象级命中明细给其他域。
- 不能输出可能被反推出本域记录存在性的细粒度统计。

### 4.2 仍可实现的联查操作

| 可实现操作 | 说明 | 边界 |
| --- | --- | --- |
| 规则包下发 | 一个域或协调方提供不含数据的规则、算子、问题清单 | 规则包不得夹带样本数据、键值、名单 |
| 本域本地计算 | 各域在自己的系统内导入本域数据并执行规则 | 计算输入、明细、中间值不出域 |
| 本域字段画像 | 识别本域字段质量、缺失、重复、格式问题 | 画像结果默认仅本域查看 |
| 本域主键 / 子键派生 | 用于本域内部表间关联、去重、校验 | 派生键不出域 |
| 本域去标识匹配 | 用于本域内部减少明文展示和内部关联 | 去标识键不出域 |
| 执行回执 | 返回“已执行 / 未执行 / 失败原因类别”等状态 | 不包含数据内容或对象级信息 |
| 结论声明 | 在授权和人工复核后返回“是否符合某事项条件”等结论 | 已确认允许输出，但不得带出明细 |
| 聚合统计 | 返回总量、符合量、不符合量、待复核量等聚合结果 | 已确认允许输出，但需满足最小阈值 |
| 人工协同 | 系统生成本域内部复核材料，由人员按制度对外答复 | 对外答复内容不由系统擅自扩大 |
| 审计证明 | 证明本域执行过某规则包、使用了哪个版本、由谁操作 | 不包含本域数据明细 |

### 4.3 待确认事项

- 是否存在协调方，只负责下发规则包而不接收数据。
- 规则包由谁定义、谁审核、如何签名和版本化。
- 每个部门是否独立部署一套系统，还是同一单位内部多个域各自部署。

### 4.4 已确认输出边界

已确认允许输出：

- 执行回执。
- 经授权结论声明。
- 聚合统计。

已确认约束：

- 聚合统计必须满足最小阈值后才能输出。
- Stage 1 先采用默认最小阈值 `10`。
- 聚合统计不得细化到可反推出单个对象或极小群体的粒度。

### 4.5 已确认治理选项

已确认采用 `BBB` 组合：

- 聚合统计最细分组口径：`方案 B`
  仅允许单维粗粒度分组，Stage 2 起仅允许按“部门、事项类型、月份”三类单一维度之一输出聚合统计，且每组都必须满足最小阈值。
- 结论声明审批流程：`方案 B`
  执行人与审核人分离，任务执行人只能生成结论草稿，审核人批准后才能输出结论声明。
- 规则包签名与导入审批：`方案 B`
  规则包必须带签名引用，并经单名审批人批准后才能用于任务执行。

## 5. 总体架构

推荐架构：多域独立单机部署 + 规则包离线流转 + 本域本地计算 + 结果本域留存。

```text
协调方或发起方
  |
  | 仅下发不含数据的规则包 / 事项包 / 算子包
  v
域 A 单机系统              域 B 单机系统              域 C 单机系统
本域数据导入              本域数据导入              本域数据导入
本域规则执行              本域规则执行              本域规则执行
本域结果留存              本域结果留存              本域结果留存
本域审计留痕              本域审计留痕              本域审计留痕
  |                         |                         |
  | 可选：执行回执/结论声明    | 可选：执行回执/结论声明    | 可选：执行回执/结论声明
  v                         v                         v
协调方或发起方
```

关键原则：

- 数据不流动，规则流动。
- 明细不流动，状态或结论按授权流动。
- 派生键不流动，内部关联能力留在本域。
- 去标识不作为跨域传输手段，只作为本域安全处理手段。

| 模块 | 功能 |

| 规则包导入 | 导入不含数据的规则包、事项包、算子包 |

| 主键 / 子键配置 | 本域内派生和验证，不提供跨域导出 |
| 去标识配置 | 本域内减少明文处理，不提供跨域导出 |

## 9. 算子库设计建议

算子库仍然需要保留，但作用从“跨域匹配”调整为“本域规则计算”。

可设计算子：

- 标准化算子：本域内部清洗字段。
- 主键 / 子键派生算子：本域内部生成关联键。
- 去标识算子：本域内部减少明文暴露。
- 条件校验算子：判断本域记录是否满足规则。
- 汇总算子：生成本域内部统计。
- 聚合阈值算子：校验统计结果是否达到最小输出阈值。
- 聚合分组约束算子：校验是否只使用单维粗粒度分组。
- 回执算子：生成无数据执行状态。
- 结论算子：根据授权生成可答复结论。

禁止或默认禁用的算子：

- 跨域集合交集算子。
- 需要导出哈希集合的匹配算子。
- 需要导出密文输入的隐私计算算子。
- 生成对象级对外明细的导出算子。

## 10. 去标识匹配设计建议

去标识匹配只能作为本域内部能力。

允许：

- 本域内部字段标准化。
- 本域内部哈希或加盐处理。
- 本域内部使用去标识键进行表间关联。
- 本域内部用去标识键减少页面明文展示。

禁止：

- 将去标识键发给其他域。
- 将哈希集合、密文集合、布隆过滤器发给其他域。
- 将去标识键作为跨域匹配协议输入。
- 将可用于验证某对象是否存在的令牌输出给对方。

## 11. 主键 / 子键派生设计建议

主键 / 子键派生只用于本域内部数据治理和任务执行。

设计原则：

- 主键用于本域内部主体关联。
- 子键用于本域内部主体下事项、时间、证照、来源等维度关联。
- 派生规则版本化。
- 派生结果不允许导出给其他域。
- 对外回执只能引用任务编号、规则包编号、执行状态，不引用主键或子键。
- 对外聚合统计只能输出满足最小阈值后的摘要值，不输出细粒度分组键。
- 结论声明必须经过审核人批准后才允许导出。

## 12. 单机离线部署建议

每个域独立部署：

- 本域前端。
- 本域后端。
- 本域数据库。
- 本域文件目录。
- 本域审计日志。

跨域流转介质只允许承载：

- 规则包。
- 算子包。
- 空模板。
- 版本清单。
- 执行回执。
- 经确认允许出域的结论声明。
- 满足最小阈值的聚合统计摘要。

跨域流转介质禁止承载：

- 原始数据。
- 加密数据。
- 哈希数据。
- 去标识数据。
- 派生主键 / 子键。
- 明细结果。
- 不满足最小阈值的统计。
- 可反推对象存在性的细粒度统计。
- 未经审批通过的规则包。

## 13. 风险与待确认项

主要风险：

- 如果结论声明过细，也可能构成数据泄露。
- 如果统计口径过细或阈值设置过低，可能反推出个体或小群体信息。
- 如果规则包夹带样例名单或键值，也会突破数据不出域约束。
- 如果审计日志记录了敏感明细，审计导出也可能变成数据出域。
- 如果前端提供任意结果导出，容易误导用户导出本域数据。
- 如果审批人与执行人未分离，结论声明的审计约束会失效。

待确认事项：

- 是否需要离线介质导入导出审批流程。
- 审核人与审批人的身份来源如何绑定到本地账号体系。

## 14. Stage 1 重新开发范围建议

重新开始 Stage 1 时，应优先实现：

1. 本域约束提示与系统模式声明。
2. 本域数据导入与字段画像。
3. 不含数据的规则包导入和列表。
4. 本域任务草稿，绑定数据集和规则包。
5. 输出策略选择：本域留存、执行回执、结论声明待审核、聚合统计输出。
6. 聚合统计最小阈值控制。
7. 规则包签名引用与单人审批骨架。
8. 审计日志记录导入、规则包导入、审批、任务创建。

Stage 1 不实现：

- 跨域数据匹配。
- 哈希集合交换。
- 加密数据交换。
- 对外明细导出。
- 完整规则引擎。
- 完整算子库。
