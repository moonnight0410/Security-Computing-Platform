# Stage 0 设计文档：政府部门联查数据计算单机系统

## 1. 项目目标与范围

本项目目标是设计一个面向政府部门联查数据场景的单机离线计算系统，用于在本地受控环境中完成数据导入、字段映射、主键或子键派生、去标识匹配、规则校验、结果生成与审计留痕。

本阶段只做设计，不进入开发实现。

系统范围限定为：

- 单机离线部署，不依赖互联网服务。
- 包含前端与后端完整方案。
- 支持结构化数据文件导入，例如表格或 CSV 类数据。
- 支持基于统一标识、组合字段、派生键、去标识键的对象匹配。
- 支持规则校验、条件判断、结果汇总和审计导出。
- 支持后续扩展算子库，但 Stage 0 不预设具体业务政策规则。

不在本阶段范围内：

- 不接入真实政务平台。
- 不虚构外部接口。
- 不预设具体部门字段标准。
- 不实现联网共享、实时调用、跨机构在线协同。
- 不开发代码或创建项目文件。

## 2. 已知约束与事实边界

已知约束：

- 系统必须是单机离线部署。
- 系统必须包含前端和后端。
- 设计必须面向后续真实开发落地。
- 需要覆盖算子库、去标识匹配、主键 / 子键派生。
- 本轮只输出 Stage 0 设计文档。

基于现有调研材料可作为设计依据的事实：

- 政府部门联查常见计算方式包括统一标识精确匹配、规则校验、共享交换、证照核验、结果反馈。
- 对敏感数据场景，去标识化、受控比对、最小化结果输出是较审慎的技术路径。
- 隐私计算、安全多方计算、联邦学习等更高级方案适合高安全或跨域协同场景，但对系统条件要求较高，不应默认作为单机系统的主方案。

事实边界：

- 现阶段未知真实使用部门、真实业务事项、真实字段清单、真实数据格式、真实政策规则。
- 现阶段不能假设存在某个外部政务接口。
- 现阶段不能假设某类字段一定存在，例如身份证号、统一社会信用代码、证照编号等，只能设计为可配置字段。
- 去标识匹配不能等同于完全匿名化，仍需按敏感数据处理。
- 单机系统只能解决本地计算与受控处理问题，不能替代跨部门授权、目录管理、共享审批和制度合规流程。

## 3. 待确认事项与关键假设

待确认事项：

- 目标用户角色：业务人员、数据管理员、审计人员、技术运维人员是否都需要使用。
- 数据来源类型：Excel、CSV、数据库导出文件、固定模板文件或其他格式。
- 数据规模：单次处理行数、字段数、文件数量、历史任务保留周期。
- 敏感字段范围：是否包含个人身份信息、企业信息、证照信息、审批信息。
- 匹配对象类型：自然人、法人、项目、事项、证照或混合对象。
- 输出要求：只输出命中结果，还是输出差异、异常、统计汇总和明细表。
- 审计要求：是否需要不可篡改日志、操作人签名、任务包归档。
- 部署环境：Windows 桌面、国产化系统、内网服务器单机或便携式工作站。

关键假设：

- 系统先按本地文件导入和本地计算设计。
- 所有字段含义、规则条件、匹配策略均由用户配置或模板导入。
- 后端负责数据处理、规则执行、审计记录和本地存储。
- 前端负责配置、预览、任务编排、结果查看和导出。
- 敏感数据默认按最小化展示和最小化导出处理。

## 4. 总体架构设计

建议采用“本地前端 + 本地后端 + 本地存储 + 本地任务引擎”的单机架构。

逻辑分层：

- 表现层：前端界面，提供数据导入、字段映射、任务配置、规则配置、结果查看。
- 应用层：后端 API，负责用户操作入口、任务管理、配置管理、权限控制。
- 计算层：匹配引擎、规则引擎、算子库、主键派生服务、去标识处理服务。
- 数据层：本地数据库、本地文件目录、任务归档目录、审计日志。
- 安全层：本地访问控制、敏感字段遮蔽、密钥或盐值管理、导出控制。

推荐运行方式：

- 前端通过本地浏览器访问。
- 后端作为本机服务运行。
- 数据文件、配置文件、任务结果均保存在本机指定工作目录。
- 不默认启用任何外部网络访问。

## 5. 前端模块设计

前端建议包含以下模块：

- 首页工作台：展示最近任务、任务状态、待处理数据、异常提醒。
- 数据导入模块：上传或选择本地文件，识别表头，预览样本数据。
- 字段映射模块：将来源表字段映射为系统内部字段角色，例如对象标识、辅助字段、校验字段、输出字段。
- 主键 / 子键配置模块：配置单字段主键、组合主键、派生子键、标准化规则。
- 去标识配置模块：配置哪些字段参与哈希、盐值策略、遮蔽展示策略。
- 匹配任务配置模块：选择数据集、匹配字段、匹配方式、命中阈值、输出字段。
- 规则校验模块：配置条件规则、比较规则、有效期规则、枚举规则、异常规则。
- 算子库管理模块：查看可用算子、参数说明、适用场景、版本。
- 结果查看模块：查看命中、未命中、冲突、异常、统计汇总。
- 审计与导出模块：导出结果、任务配置、日志摘要、处理报告。

前端设计重点：

- 避免直接暴露敏感字段全文。
- 所有任务配置应可保存、复用、导入和导出。
- 规则配置要可解释，避免黑盒判断。
- 结果页面应明确区分“系统计算结果”和“业务最终结论”。

## 6. 后端模块设计

后端建议包含以下模块：

- 文件解析模块：读取 Excel、CSV 等结构化文件，完成表头识别、编码处理、类型初判。
- 数据画像模块：统计空值率、重复率、字段类型、样本值、潜在主键候选。
- 字段映射模块：保存字段角色、字段别名、标准化方式。
- 主键派生模块：根据配置生成主键、组合键、子键、去标识键。
- 去标识处理模块：对敏感字段进行标准化、哈希、遮蔽、最小化输出。
- 匹配引擎模块：执行精确匹配、组合匹配、规则约束匹配、可选近似匹配。
- 规则引擎模块：执行条件判断、区间判断、日期有效性判断、字段一致性判断。
- 算子库模块：统一管理计算算子、参数校验、版本信息和执行日志。
- 任务调度模块：创建任务、运行任务、暂停任务、记录进度、处理失败重试。
- 本地存储模块：管理数据集、配置、任务结果、审计日志。
- 导出模块：生成结果文件、统计报告、审计包。
- 安全模块：本地权限、敏感字段访问控制、导出控制、密钥配置。

## 7. 数据流与核心数据模型建议

典型数据流：

1. 用户导入本地数据文件。
2. 系统解析字段并生成数据画像。
3. 用户确认字段映射和敏感字段标记。
4. 用户配置主键 / 子键派生规则。
5. 系统生成标准化字段、派生键或去标识键。
6. 用户配置匹配任务和校验规则。
7. 后端执行匹配与规则计算。
8. 系统生成命中结果、异常结果和统计汇总。
9. 用户查看、复核并导出结果与审计记录。

核心数据模型建议：

- 数据集 Dataset：名称、来源文件、导入时间、字段清单、行数、数据摘要。
- 字段 FieldProfile：字段名、类型推断、空值率、重复率、敏感级别、字段角色。
- 映射配置 FieldMapping：来源字段、内部角色、标准化策略、是否输出。
- 主键配置 KeyRule：主键类型、参与字段、清洗规则、拼接规则、派生算法。
- 去标识配置 DeIdRule：参与字段、盐值策略、哈希算法、遮蔽策略。
- 匹配任务 MatchTask：任务名称、左表、右表、匹配规则、输出策略、状态。
- 规则配置 ValidationRule：条件表达式、算子、参数、严重级别、提示文本。
- 计算结果 MatchResult：匹配状态、匹配键、命中对象、差异字段、规则结果。
- 审计日志 AuditLog：操作人、操作时间、操作类型、配置摘要、结果摘要。

## 8. 算子库设计建议

算子库应作为后端核心能力之一，面向“可配置、可解释、可审计”设计。

建议算子分类：

- 标准化算子：去空格、大小写统一、全半角转换、日期格式转换、证件号格式清洗。
- 精确匹配算子：单字段等值匹配、组合字段等值匹配、派生键匹配。
- 集合算子：包含、不包含、交集、差集、重复检测。
- 比较算子：等于、不等于、大于、小于、区间判断。
- 日期算子：有效期判断、起止时间比较、年龄或期限计算。
- 规则算子：条件组合、全部满足、任一满足、互斥判断。
- 统计算子：计数、去重计数、分组汇总、比例计算。
- 异常识别算子：空值异常、重复异常、字段冲突、格式异常。
- 可选近似算子：名称相似度、地址文本相似度，但应默认作为辅助提示，不作为最终事实判断。

算子设计原则：

- 每个算子需要有名称、版本、输入参数、输出结构、适用场景、限制说明。
- 算子执行结果应保留可解释信息。
- 涉及敏感字段的算子应支持只输出布尔结果或摘要结果。
- 不应将业务政策固化进基础算子，业务规则应通过配置组合实现。

## 9. 去标识匹配设计建议

去标识匹配目标是在不直接展示原始敏感标识的前提下完成对象关联或比对。

建议流程：

1. 字段标准化：对参与字段进行格式清洗，减少同一对象因格式差异导致无法匹配。
2. 字段组合：按配置选择一个或多个字段组成匹配输入。
3. 加盐或密钥处理：使用本地配置的盐值或密钥增强抗反推能力。
4. 哈希生成：生成不可直接读出原始值的去标识键。
5. 匹配计算：使用去标识键进行等值匹配或集合比对。
6. 最小化输出：默认只输出命中状态、规则结果、必要摘要，不输出原始敏感值。
7. 审计记录：记录使用了哪些配置版本，不记录明文敏感输入。

边界说明：

- 去标识键仍可能在特定条件下被重识别，不能当作完全匿名数据。
- 如果不同任务使用相同盐值，可能形成跨任务关联能力，需要管理复用范围。
- 如果参与字段质量差，去标识匹配仍会产生漏匹配或错匹配。
- 去标识匹配适合本地受控比对，不等同于跨部门在线隐私计算。

## 10. 主键 / 子键派生设计建议

主键 / 子键派生用于将原始数据中的一个或多个字段转换为系统可稳定使用的对象标识。

主键设计：

- 优先使用业务上明确唯一、稳定、可验证的字段。
- 如果没有单一主键，可使用组合主键。
- 主键生成前必须进行标准化处理。
- 主键规则必须可查看、可复用、可审计。

子键设计：

- 子键用于表达同一对象下的局部维度，例如对象 + 时间、对象 + 事项、对象 + 证照。
- 子键不应替代主键，而应作为细分匹配或规则校验依据。
- 子键可用于发现同一主体下的多条记录、重复记录、冲突记录。

派生方式建议：

- 单字段派生：标准化后直接作为键。
- 组合字段派生：多个字段按固定顺序拼接后生成键。
- 命名空间派生：在键前加入对象类型或数据集标识，避免不同业务域键冲突。
- 哈希派生：对标准化后的键材料生成摘要，减少敏感值暴露。
- 版本化派生：同一规则变更后生成新版本，避免历史任务不可追溯。

风险控制：

- 派生规则变更会影响匹配结果，必须记录版本。
- 不同字段缺失时应明确处理策略，例如拒绝生成、生成弱键、进入异常队列。
- 弱唯一字段不应单独作为主键，只能作为辅助匹配字段。

## 11. 单机离线部署建议

建议部署形态：

- 前端：本地 Web UI。
- 后端：本机 API 服务。
- 存储：本地数据库 + 本地文件目录。
- 运行环境：单机内网或离线电脑。
- 数据目录：按任务隔离原始文件、处理中间文件、结果文件、审计文件。

安全建议：

- 默认关闭外网访问。
- 后端只监听本机地址或受控内网地址。
- 敏感字段默认遮蔽展示。
- 导出结果应支持字段白名单。
- 本地配置、盐值、密钥应与任务数据分开保存。
- 任务归档应包含配置摘要、算子版本、运行时间和结果摘要。
- 删除数据时应区分删除任务记录、删除结果、删除原始导入文件。

运维建议：

- 提供一键启动和停止方式。
- 提供本地日志查看。
- 提供备份和恢复机制。
- 支持离线升级包，但升级前应备份配置和任务数据。
- 对大文件处理应提供进度、失败原因和可恢复机制。

## 12. Stage 1 开发建议

Stage 1 建议只实现最小可用闭环，不一次性实现所有复杂能力。

建议优先级：

1. 本地前后端基础框架。
2. 文件导入、字段预览、字段画像。
3. 字段映射与配置保存。
4. 主键 / 组合键派生。
5. 精确匹配任务执行。
6. 基础规则校验。
7. 结果查看与导出。
8. 审计日志。
9. 去标识键生成与匹配。
10. 算子库版本化管理。

Stage 1 不建议优先实现：

- 复杂可视化大屏。
- 在线政务接口接入。
- 分布式计算。
- 安全多方计算。
- 联邦学习。
- 未确认业务规则的硬编码模板。
